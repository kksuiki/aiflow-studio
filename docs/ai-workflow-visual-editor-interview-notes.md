# 前端面试项目准备文档（AI 工作流可视化编辑器）

本文按“围绕项目、深挖思路”的面试方法来组织，目标不是背答案，而是让你能连续回答四层追问：

1. 你做了什么？
2. 你为什么这么做？
3. 如果不用这个库/方案，你怎么做？
4. 你知道它底层原理吗？

技术背景：`React 18 + TypeScript + React Flow + Zustand + Ant Design + SSE`

---

## 一、准备原则：围绕项目，深挖思路

项目面试的关键不是“功能有多炫”，而是你是否能把每个关键点讲清楚：

1. 目标是什么（业务价值）
2. 前端是如何拆层实现的（页面、组件、状态、接口）
3. 为什么这样选型（对比和取舍）
4. 有没有边界问题和优化动作（性能、可维护、可观测）

你当前项目最强主线建议：

1. **AI 工作流可视化编排（图编辑器）**
2. **复杂状态管理（节点/边/执行态一致性）**
3. **流式调试可观测（SSE 节点级回传）**
4. **RAG 节点能力接入（知识库到运行链路）**

## 二、准备内容结构（按功能点逐块拆解）

---

### 功能点 A：可视化工作流编排（节点拖拽、连线、缩放）

#### 1. 实现目标
让用户像搭积木一样通过拖拽节点和连线来构建 AI 工作流，而不是写代码。

#### 2. 前端组织方式
1. 节点面板：提供可拖拽的节点类型（LLM/RAG/Condition/Skill 等）。
2. 画布：承载节点渲染、连线渲染、缩放/平移、MiniMap。
3. 交互事件：`onDrop / onConnect / onNodesChange / onEdgesChange`。
4. 数据结构：以 `nodes + edges` 作为工作流核心状态。

#### 3. 状态管理方式
Zustand 统一存储 `nodes/edges/selectedNode/currentWorkflow`，交互事件直接回流到 store。

#### 4. 与后端对接
保存时提交 `nodes + edges`；加载时还原图结构到画布。

#### 5. 性能/体验考虑
1. 图编辑器能力由 React Flow 承担，避免自研底层坐标系统与连接线系统。
2. 通过视图自适应、MiniMap、状态图标提升复杂流程可读性。

#### 6. 口头简明版（30 秒）
“我们把工作流抽象成有向图，前端维护 `nodes + edges`。用户从节点面板拖拽节点到画布，通过锚点连线建立依赖。所有交互事件统一回流到 Zustand，保证画布和配置状态一致，最后序列化保存到后端。”

#### 7. 深入问答版（90 秒）
“可视化编排的核心不是 UI，而是图状态一致性。我们用 React Flow 管理拖拽、连线、缩放、坐标转换，避免自研复杂底层。业务层通过 Zustand 持有 `nodes/edges`，`onConnect` 时写入 edge，`onDrop` 时创建节点默认数据。这样任意时刻工作流都可序列化持久化，且便于后续执行引擎消费。”

---

### 功能点 B：多类型节点 + 节点参数配置

#### 1. 实现目标
不同节点具备不同业务语义和参数，支持无代码配置执行逻辑。

#### 2. 前端组织方式
1. 节点类型固定：`start/userInput/llm/rag/skill/condition/output`。
2. 右侧配置面板根据 `selectedNode.type` 动态渲染表单。
3. 表单修改时实时写回节点 `data`。

#### 3. 状态管理方式
1. `selectedNode` 记录当前编辑对象。
2. `updateNodeData(nodeId, data)` 同步更新节点和选中态，防止面板与画布数据不一致。

#### 4. 与后端对接
配置结果最终和 `nodes + edges` 一起保存；运行时后端按节点类型执行。

#### 5. 校验与体验
1. 必填字段校验（如 userInput 的输入字段、RAG 的 knowledgeBaseId）。
2. JSON 文本输入（skill 参数、condition 规则）提供格式约束。

#### 6. 口头简明版（30 秒）
“我们做了类型化节点模型，不同节点有不同 data 结构。配置面板是动态表单，跟随选中节点类型切换，变更后实时写回 store，保证配置即状态。”

#### 7. 深入问答版（90 秒）
“关键是‘动态表单 + 类型化状态’。节点类型定义了字段边界，比如 LLM 节点有模型参数，RAG 节点有知识库与检索参数。配置面板通过 `selectedNode.type` 渲染对应表单，`onValuesChange` 调 `updateNodeData`。这样既能做节点差异化，又能保持统一保存结构。”

---

### 功能点 C：工作流调试与运行（SSE 节点级可观测）

#### 1. 实现目标
不仅要有最终结果，还要能看到每个节点执行状态和输出，便于排障。

#### 2. 前端组织方式
1. RunPanel 输入运行参数（JSON）。
2. 发起 `/workflows/:id/run/stream`。
3. 解析流式事件：`node_status / done / error`。
4. 将每条节点事件写入 `executionStates[nodeId]`。
5. 画布节点与调试面板同步展示运行状态和输出。

#### 3. 状态管理方式
1. `executionStatus`：全局运行状态（running/success/failed/stopped）。
2. `executionStates`：节点级状态映射，支持增量更新。

#### 4. 为什么是 SSE
当前需求是服务端单向推送进度流，SSE 比 WebSocket 更轻量、落地更快、调试更直观。

#### 5. 口头简明版（30 秒）
“运行阶段我们用 SSE 做流式回传。后端每完成一个节点就推一条 `node_status`，前端按 `nodeId` 更新执行状态，所以能实时看到哪个节点在跑、哪个失败、输出是什么。”

#### 6. 深入问答版（90 秒）
“SSE 返回的是事件流，不是一次性响应。我们拿到 reader 后用 parser 持续消费事件，`node_status` 就写入 `executionStates`，`done/error` 更新全局状态。UI 分两层渲染：节点组件显示状态图标，RunPanel 展示节点输出/错误。这套机制把工作流执行从黑盒变成可观测流水线。”

---

### 功能点 D：工作流管理（保存/加载/删除）

#### 1. 实现目标
让编辑器从“临时画布”变成“可复用资产管理”。

#### 2. 前端组织方式
1. 应用列表进入编辑器，按应用加载流程。
2. 保存时提交当前图结构。
3. 首次无流程时自动创建默认流程，提高可用性。

#### 3. 状态与接口
1. `fetchWorkflows / fetchWorkflowById / createWorkflow / saveWorkflow / deleteWorkflow`
2. 全部通过切片 action 统一封装。

#### 4. 口头简明版（30 秒）
“我们把工作流当作可持久化资源管理，支持创建、加载、保存、删除，并在空态时自动生成默认流程降低上手门槛。”

## 三、技术选型（为什么用这个框架/库）

---

### 1) 为什么用 React Flow，而不是自己写拖拽？

#### 结论
复用成熟图编辑引擎，把研发重心放在 AI 工作流业务语义与执行链路。

#### 取舍对比
1. React Flow 优势：拖拽、连线、坐标转换、缩放、选中、MiniMap、事件回调生态完整。
2. 自研成本：坐标系统、碰撞检测、曲线连线、性能优化、事件冲突处理工作量大。
3. 项目收益：更快交付核心业务能力（节点配置、调试、状态一致性）。

#### 被追问“如果不用 React Flow？”
可回答：
1. 绝对定位渲染节点 + SVG 渲染边。
2. 手写 drag/drop、连接锚点、缩放平移。
3. 封装图状态管理与撤销重做。
4. 但生产项目优先复用成熟库降低风险。

---

### 2) 为什么用 Zustand，而不是 Redux/Context API？

#### 结论
当前项目需要“低样板 + 细粒度订阅 + 切片治理”，Zustand 是更平衡的选择。

#### 取舍对比
1. 对比 Redux：规范强但样板偏多、迭代成本更高。
2. 对比 Context：状态复杂时容易触发不必要重渲染，拆分和维护成本上升。
3. Zustand：切片组织自然，action 写法直接，选择器订阅减少渲染范围。

#### 被追问“如果不用 Zustand？”
可回答：
1. Redux Toolkit：`slice + thunk` 做标准化状态流。
2. Context + useReducer：小项目可行，大项目需拆分多 Provider 并优化渲染。
3. 自建 store：发布订阅模型可实现，但需要自行维护中间件能力。

---

### 3) 为什么用 Ant Design？

#### 结论
中后台表单/表格/弹窗密集场景，AntD 能显著降低 UI 研发成本并保持交互一致性。

#### 取舍对比
1. AntD：组件完整、稳定、表单能力强、团队维护成本低。
2. 其他方案：可能更轻或更自由，但在复杂后台交互上要补大量基础能力。

---

### 4) 为什么用 SSE，而不是 WebSocket？

#### 结论
场景核心是“服务端到前端单向推送执行进度”，SSE 更轻、足够、实现快。

#### 取舍对比
1. SSE：单向、基于 HTTP、调试简单、部署门槛低。
2. WebSocket：双向能力更强，但当前需求不需要，复杂度更高。
3. 轮询：实现简单但延迟高、请求浪费明显。

#### 被追问“如果不用 SSE？”
可回答：
1. WebSocket 推节点事件。
2. Task + 轮询查询进度。
3. 长轮询作为低配替代。

## 四、自主实现思路（假设不用库）

---

### 1) 不用 React Flow，怎么实现简易拖拽连线编辑器？

1. 图模型定义：`nodes/edges/viewport`
2. 渲染：节点 DOM + 边 SVG
3. 交互：拖拽节点、锚点连线、缩放平移
4. 状态同步：选中态、连线预览态、撤销重做
5. 持久化：序列化保存/加载

可补一句：
“从工程角度看，真正难的是交互边界和性能稳定性，而不是把节点画出来。”

### 2) 不用 Zustand，怎么管理复杂全局状态？

1. Redux Toolkit：规范化 action/reducer/thunk
2. Context 分域 + useReducer + memo + selector
3. 统一接口层 + 状态机约束执行生命周期

### 3) 不用 SSE，怎么做实时回传？

1. WebSocket 双向通道
2. 后端任务队列 + 前端轮询
3. 事件持久化日志 + 增量拉取

## 五、原理讲解与底层理解（加分项）

---

### 1) React Flow 核心原理（你可以这样讲）
“React Flow 把图编辑器抽象成受控状态系统。节点和边由数据驱动渲染，坐标转换负责把屏幕交互映射到画布坐标。`onNodesChange/onEdgesChange/onConnect` 等事件把用户操作回流到状态层，状态变更再驱动 UI 更新。”

### 2) Zustand 原理（你可以这样讲）
“Zustand 本质是外部 store + 发布订阅。组件通过 selector 订阅局部状态，只有订阅片段变化才重渲染，所以相比 Context API 的全量传播更高效，特别适合图编辑器这种高频局部更新场景。”

### 3) SSE 原理（你可以这样讲）
“SSE 是基于 HTTP 长连接的事件流，服务端按 `text/event-stream` 持续推送事件。前端持续读取并解析事件，按事件类型更新状态。它天然适合单向进度推送，代码和部署复杂度都低于 WebSocket。”

“我这个项目核心是 AI 工作流可视化编排，不是普通 CRUD。前端基于 React Flow 实现节点拖拽、连线和多类型节点配置，用 Zustand 管理 nodes/edges/执行态等复杂状态，保证画布与配置面板一致。运行阶段通过 SSE 做节点级状态回传和可观测调试，能实时看到每个节点的执行结果和错误。整体上兼顾了低代码编排效率、调试效率和可维护性。”
